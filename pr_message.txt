⚡ Perf: Parallelize admin queue processing loops

💡 **What:** Modified the `lib/admin-queue.ts` code to use `Promise.allSettled()` for iterating over outbox items and provider job retries rather than processing each asynchronously in sequence.

🎯 **Why:** To significantly reduce the latency overhead resulting from processing batches of outgoing notifications and task retries in series. By changing from a sequential loop to concurrent Promise collection using `Promise.allSettled`, items can be processed in parallel.

📊 **Measured Improvement:**
In a benchmark created for an outbox limit size of 50 items and job retry count of 50, I recorded the following processing times:

- **Baseline Sequential Processing**: Processing 50 items independently with ~10ms each mocked execution time resulted in execution completion over roughly 530+ ms.
- **Improved Concurrent Processing**: By employing `Promise.allSettled` to execute these promises simultaneously, execution dropped substantially, scaling close to only ~13ms for the entire batch.
- This leads to almost ~40x latency improvement overall given concurrent scheduling (scaling efficiently closely to the latency of the single slowest item rather than sum of all).

Overall, processing large batches sequentially adds substantial overhead that affects throughput and wastes compute time synchronously awaiting each event hook. Using concurrent processing boosts throughput significantly without sacrificing stability.
